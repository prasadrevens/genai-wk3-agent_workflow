# ImpactIQ Project Documentation

## Project Overview

ImpactIQ - Business-Aware AIOps is an end-to-end incident triage demo that combines:

- A live Mini Shop FastAPI application that generates demo telemetry.
- A LangGraph multi-agent AIOps workflow.
- A React dashboard for incident monitoring, investigation, RCA, and gated action.
- A Phase 1 and Phase 2 Voice Incident Commander experience.

The purpose of the project is to show how an AI incident workflow can connect technical evidence to business impact without allowing the AI to make uncontrolled production changes.

The core workflow is:

```text
Mini Shop telemetry
  -> ImpactIQ FastAPI dashboard APIs
  -> React dashboard
  -> LangGraph Commander-led investigation
  -> Business Impact
  -> RCA
  -> Human Approval Gate
```

The LangGraph architecture follows a commander-led supervisor pattern:

```text
Metrics  \
Logs      \
Trace      -> Commander -> Business Impact -> RCA -> Human Gate
Changes   /
```

The specialist agents do not directly talk to each other. They provide findings to Commander, and Commander decides when enough evidence has been collected to move to business impact and RCA.

## Datasets Used

This project does not use an external public dataset. It uses synthetic but realistic telemetry generated by the Mini Shop demo application.

The generated telemetry files are:

```text
mini_shop_with_ui/data/alert.json
mini_shop_with_ui/data/logs.json
mini_shop_with_ui/data/metrics.json
mini_shop_with_ui/data/traces.json
mini_shop_with_ui/data/changes.json
mini_shop_with_ui/data/business_metrics.json
mini_shop_with_ui/data/service_dependencies.json
mini_shop_with_ui/data/state.json
```

### `alert.json`

Represents the incident alert.

Example meaning:

- Incident ID
- Timestamp
- Severity
- Affected service
- Business workflow
- Alert title
- Symptoms

### `logs.json`

Application log events.

Logs answer:

- What error messages happened?
- Which service emitted them?
- Are repeated failures visible?
- Did checkout or payment start timing out?

Example signal:

```text
Checkout failed due to payment timeout
```

### `metrics.json`

Numeric measurements over time.

Metrics answer:

- Is latency increasing?
- Are failures increasing?
- How severe is the issue?

Example metrics:

```text
checkout.latency
payment.latency
orders_db.latency
checkout.failure
deployment.event
```

### `traces.json`

Distributed trace-like request paths.

Traces answer:

- Which service is slow?
- Where is the bottleneck?
- Did a checkout request fail?
- What was the critical request path?

Example checkout path:

```text
frontend -> checkout-api -> payment-api -> orders-db -> notification-service
```

### `changes.json`

Recent deployment/change events.

Changes answer:

- Was there a recent deployment?
- Did the timing line up with the incident?
- Which service changed?

Example change:

```text
checkout-api deployed v1.0.1-bad
```

### `business_metrics.json`

Business impact metrics.

Business metrics answer:

- How many transactions happened?
- How many failed?
- What is the payment success rate?
- What is the estimated revenue impact?

Important fields:

```text
transactions_last_24h
failed_transactions_last_24h
payment_success_rate_percent
revenue_last_24h
estimated_revenue_impact
business_summary
```

### `service_dependencies.json`

Dependency map for the checkout workflow.

It powers the service dependency tree in the dashboard.

Example critical path:

```text
frontend -> checkout-api -> payment-api -> orders-db -> notification-service
```

### `state.json`

Internal Mini Shop app state.

It tracks values such as:

- Current deploy mode: `good` or `bad`
- Deployment version
- Transaction count
- Failed transaction count
- Revenue

## Iterations Tried

### Iteration 1: Streamlit UI Repair

The first UI work focused on Streamlit layout issues where card content appeared outside the card container.

Learning:

- Layout bugs can make a working backend look broken.
- For investigation/RCA panels, fixed height plus internal scrolling improves demo usability.

### Iteration 2: Static HTML To React Dashboard

The initial dashboard design was converted from static HTML into React components.

Learning:

- Clear component ownership made state easier to manage.
- `SentinelDashboard` became the parent for incident, RCA, timeline, pipeline, loading, and error state.

### Iteration 3: Backend Endpoint Scaffolding

FastAPI endpoints were added or adapted for the dashboard:

- `/api/incident`
- `/api/rca`
- `/api/triage/run`
- `/api/triage/stream`
- `/api/triage/decision`
- `/api/triage/confirm-rollback`

Learning:

- A stable UI contract is important even if backend telemetry sources change.
- SSE works well for streaming investigation timeline events.

### Iteration 4: Mini Shop Telemetry Integration

The agent moved from static incident fixtures to live Mini Shop telemetry files.

Learning:

- The demo became more realistic once telemetry was generated by an app flow.
- Missing telemetry files produced degraded RCA, which made the setup sequence important.

### Iteration 5: Automatic Telemetry Polling

The dashboard was updated to poll incident telemetry approximately every 5 seconds.

Learning:

- This removed a manual "reload telemetry" step.
- Live polling made the alert panel and KPI cards feel operational.

### Iteration 6: Healthy vs Degraded Visual Accuracy

The service dependency tree originally showed bottleneck/timeout states even when the incident was healthy.

Learning:

- Visual state must be derived from incident state.
- Healthy incidents should render services as OK.

### Iteration 7: Human Approval Guardrails

Approval and reject controls were disabled while triage was running and only unlocked after the human gate.

Learning:

- Human-in-the-loop controls should mirror backend workflow state.
- The UI should not imply that remediation can happen before RCA is complete.

### Iteration 8: Architecture Review And Workflow Visualization

The linear pipeline visualization was reviewed against the actual LangGraph code.

Observation:

- The actual workflow is not a simple linear pipeline.
- It is a commander-led supervisor pattern.

Learning:

- The visualization should show specialists feeding Commander, then Commander passing evidence to Business Impact and RCA.
- A misleading visualization can cause users to misunderstand the agent architecture.

### Iteration 9: Agent Workflow Visual Design

Several versions of the agent workflow map were tried:

- Linear strip.
- Two-panel map.
- Commander-centered map.
- Box-based flow with agent logos and color-coded states.

Learning:

- A single consistent left-to-right flow was easiest to understand:

```text
Metrics / Logs / Trace / Changes -> Commander -> Business Impact -> RCA
```

- Arrows must connect cleanly to boxes; even small alignment issues make the diagram feel broken.

### Iteration 10: Business Metrics Consistency

Payment success and estimated impact colors were aligned between the top KPI cards and the Business Metrics panel.

Learning:

- Users notice color inconsistency quickly.
- Thresholds must be reused across views.

### Iteration 11: Payment Success Calculation Fix

Payment success briefly appeared as a negative number.

Root cause:

- Failed transactions were being subtracted from successful transactions even though the state already tracked successful checkouts separately.

Fix:

- Calculate success as:

```text
successful_transactions / (successful_transactions + failed_transactions)
```

Learning:

- Demo telemetry counters need clear semantic definitions.
- UI-level clamping is useful, but the source calculation should also be correct.

### Iteration 12: Voice Incident Commander

Phase 1 and Phase 2 were implemented:

- Suggested voice commands.
- Simulated transcript input.
- Text-only answer function.
- No ElevenLabs call.
- No remediation trigger.

Learning:

- Voice should be layered on top of dashboard state first.
- Voice interaction must not bypass approval gates.

### Iteration 13: GitHub Cleanup And Push

## Learnings And Observations

### 1. The Agent Architecture Should Be Verified In Code

The initial UI implied a linear pipeline:

```text
Commander -> Metrics -> Logs -> Trace -> Changes -> Business -> RCA
```

But the code showed a supervisor pattern:

```text
Specialists -> Commander -> Business -> RCA
```

This was an important correction because the UI should teach the real architecture.

### 2. Business-Aware AIOps Needs Business Metrics As First-Class Inputs

Technical telemetry alone can identify symptoms, but business metrics explain impact.

The most useful business fields were:

- Payment success rate
- Failed transactions
- Revenue impact
- Total transactions

### 3. Tool Failures Should Degrade, Not Crash

When telemetry files are missing, tools should report degraded findings instead of breaking the workflow.

This made it easier to explain why the Mini Shop demo flow must be run before triage.

### 4. The UI Must Match Incident State

Healthy state should look healthy:

- Green status
- OK dependencies
- Healthy payment success

Degraded state should highlight only the affected areas.

### 5. Human Approval Must Stay Explicit

The system can recommend remediation, but approval must remain explicit.

The final design keeps this sequence:

```text
RCA recommendation -> Approve -> second confirmation -> rollback integration
```

### 6. Voice Assistant Should Start As Read-Only

The Voice Incident Commander was intentionally implemented as a read-only text assistant first.

This gives value without creating risk:

- It can answer incident questions.
- It cannot trigger remediation.
- It does not call ElevenLabs yet.

### 7. Demo UX Matters As Much As Backend Correctness

Many iterations were visual and interaction-focused:

- Card heights.
- Internal scrolling.
- Color thresholds.
- Workflow arrows.
- Brand naming.
- Button placement.

These changes made the system easier to explain and more credible in a live demo.
